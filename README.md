# 问卷反馈双站点

- 用户侧网站：`http://localhost:5011`
- 管理侧网站：`http://localhost:5012`
- 管理员账号：`cyx` / `jy`
- 管理员密码：`asdf`

## 启动

```bash
npm install
npm start
```

## 功能概览

- 用户侧：优雅粉紫风首页、Bug 反馈、多条功能建议反馈、图片上传、管理员码识别。
- 管理侧：管理员登录、Bug 管理、建议管理、筛选编辑删除、智能需求文档生成、主题色与字体调节预览。
- 配置：编辑 `config.cfg` 可修改端口、管理员列表、主题色、字体方案与大模型接口。

## 当前技术方案

- 前端：原生 `HTML + CSS + ES Modules`，没有引入 React / Vue 等前端框架，方便快速落地和直接改样式。
- 后端：`Node.js + Express`，当前不是 `MongoDB + FastAPI`，也不是前后端完全分离的多服务方案。
- 站点结构：同一个 Node 进程里启动两个 Express 站点：
  - 用户侧：`5011`
  - 管理侧：`5012`
- 图片上传：使用 `multer` 处理上传，图片直接写入本地 `uploads/`
- 管理登录：使用 `express-session` 维护管理员登录态
- 智能文档生成：后端直接读取 `config.cfg` 中配置的大模型接口，按顺序 fallback，请求前一个失败后再尝试下一个

## 当前数据存储方案

- 当前不是数据库方案，而是“本地 JSON 文件 + 本地文件目录”的轻量实现：
  - `data/bugs.json`：存储 Bug 反馈
  - `data/suggestions.json`：存储功能建议 / 其他反馈
  - `data/reports.json`：存储历史生成的需求文档和任务结果
  - `uploads/`：存储用户上传的图片
  - `config.cfg`：存储管理员、端口、主题、字体、大模型接口等配置
- 这个方案的优点是实现简单、迁移容易、开发速度快，适合当前先把产品流程跑通。

## 当前方案未来可能遇到的问题

- 并发写入风险：多个请求同时修改 JSON 文件时，未来会有写覆盖、数据竞争、文件损坏风险。
- 不适合多实例部署：如果未来要部署到多台机器，本地 JSON 和本地上传目录无法天然共享。
- 会话存储过轻：当前 `express-session` 默认内存会话更适合开发阶段，不适合长期生产运行。
- 报告任务不够稳定：智能文档生成目前是进程内后台任务，如果服务重启或异常退出，任务可能中断。
- 安全性不足：管理员密码和大模型 API Key 目前写在 `config.cfg`，开发方便，但正式环境不够安全。
- 检索能力有限：随着反馈数量增加，JSON 文件在筛选、分页、统计、审计方面会越来越吃力。
- 文件管理压力会上升：图片多了之后，本地磁盘清理、备份、迁移、访问性能都会变成问题。

## 建议的后续迭代和优化

- 数据层升级：
  - 小规模继续演进可以先切到 `SQLite`
  - 更正式的长期方案建议改成 `PostgreSQL` 或 `MongoDB`
- 会话和认证升级：
  - 管理员密码改成哈希存储
  - 会话持久化到 `Redis` 或数据库
  - 增加更细的管理员权限、操作日志、审计记录
- 智能任务升级：
  - 把“生成需求文档”改成真正的任务队列，例如 `Redis + BullMQ`
  - 增加任务重试、失败记录、生成状态追踪、历史版本管理
- 文件存储升级：
  - 图片迁移到对象存储，例如 OSS / COS / S3
  - 增加压缩、缩略图、容量控制、清理策略
- 工程化升级：
  - 增加请求参数校验、错误监控、结构化日志、自动化测试
  - 增加分页、排序、批量处理、导出能力
  - 进一步规范 API 结构，方便后续拆成独立前后端

## 如果未来要升级成更正式的生产方案

- 一个稳妥的演进方向可以是：
  - 后端：继续用 `Node.js + Express`，或者切换到 `FastAPI`
  - 数据库：`PostgreSQL` / `MongoDB`
  - 会话和队列：`Redis`
  - 文件：对象存储
  - 网关：`Nginx`
- 当前这版更适合：
  - 快速原型
  - 小范围内测
  - 先验证真实反馈流程和管理流程
